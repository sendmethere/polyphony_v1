# Socratic Planning Log — Polyphony (대화주의 기반 토론 오케스트레이션 도구)
_Started: 2026-06-22_

## 배경
- `paper/` 의 대화주의(dialogism) 문헌 2편 정독 (Wegerif, *Dialogic*; Trausan-Matu·Wegerif·Major, *Dialogism*).
- 핵심 철학: 의미는 "차이의 틈"에서 발생 / 폴리포니엔 우세·권위적 목소리가 없음 / 발산↔수렴 반복 / 대화공간의 확장이 목표.

## 한 줄 개념
질문 제시 → 학습자 의견 수집 → (AI) 의견 군집화 → (AI) 군집을 세분/통합하는 후속 질문 제안 → 다시 수집 → 군집화 → … 의 사이클을, **가로 스크롤**로 전개를 시각화하며 웹에서 모니터링·오케스트레이션하는 프로토타입.
- 웹 기반, 결과는 **localStorage** 저장.
- **API KEY**를 설정에서 입력 가능.
- 목적: 대화의 **다성성(polyphony)** 시각화. **우세한 의견이 드러나면 안 됨** — 최대한 다양한 의견을 담는 것.
- 화면 3종: **학생 / 교사(모니터링) / 관리자(세션 진행)**.

---

## Q1 — 시각화 원칙: 우세 의견을 어떻게 감출 것인가
**Claude:** 군집화 시 "큰 군집 = 다수 = 이긴 의견"으로 보이는 문제. 옵션 (1)크기무시 동등카드 (2)색/질감으로만 (3)외톨이 의견 강조 (4)군집 간 긴장선 강조.
**User:** 크기 무시, 색·질감도 모두 동일하게. 단, **관리자 화면에서는** (내부 데이터를) 볼 수 있게.
**Takeaway:** 학생·교사 화면 = 완전 동등 표현(우세 은폐). 관리자 화면 = 군집 크기/카운트 등 내부 지표 열람 가능. "무대"와 "무대 뒤" 분리.

## Q2 — AI vs 관리자 역할 분담
**Claude:** 군집화/후속질문 생성에서 AI 자동 vs 사람 개입 수준? 세분/통합 방향은 누가 정하나?
**User:** AI가 제안하면 관리자(교사)가 선택해서 진행.
**Takeaway:** AI = 조력자(군집안 + 세분/통합 후속질문 후보 제시). 관리자(교사) = 선택·확정으로 진행. AI는 권위적 결론을 내리지 않음(비-monologic). ※ '관리자(교사)'로 묶어 표현 — 교사/관리자 역할 분리 여부는 추후 확인 필요.

## Q3 — 가로 스크롤 시각화 구조
**Claude:** 라운드=열(시간 좌→우) vs 계보/분기 흐름 vs 타임라인 vs 무한캔버스?
**User:** (1) 열 구조 기본 + 강물/계보 분기 뷰로도 볼 수 있게.
**Takeaway:** 메인 뷰 = 라운드별 세로 열, 시간 좌→우 진행. 토글로 계보/강물(분기·합류 직조) 뷰 전환. 열 사이 연결선 = 군집의 세분(갈라짐)/통합(합류).

## Q4 — 참여 방식 & 백엔드 (일부)
**Claude:** 단일 기기 프로토타입인가, 여러 학생 실시간 접속인가? 학생 입력 위치/참가 방식은?
**User:** 실시간 접속 전제. localStorage 대신 Firebase 필요.
**Takeaway:** **아키텍처 변경** — 실시간 다중 기기. 동기화 백엔드 = Firebase(Firestore/RTDB). 화면 3종이 같은 세션을 실시간 구독. localStorage는 세팅/시크릿(API KEY) 보관 + JSON 내보내기 보조 용도로 재배치. (학생 입력 위치 A / 참가 방식 B는 후속 확인.)

## Q5 — 학생 화면 & 참가 방식
**Claude:** 학생 화면 노출 범위(A: 순수입력/히스토리/군집반응) + 참가 방식(B: 코드+닉/익명/익명표시)?
**User:** A는 (1) 기본 + 관리자가 (3) ON 가능. B는 (3).
**Takeaway:** 학생 기본 = 순수 입력(현재 질문+입력칸). 관리자 토글로 "군집 열람+반응(interanimation)" 모드 개방 가능. 참가 = 세션코드+닉네임 입장, 무대 표시는 익명.

## Q6 — 교사 vs 관리자 역할 경계
**Claude:** 별개 역할 / 같은 사람 다른 뷰 / 위계형 중? 모니터링 지표는?
**User:** 연구에 따라 다르지만 (2)로 가자.
**Takeaway:** **(2) 같은 사람, 다른 뷰.** 진행자=동일 권한, "진행 모드(관리자 화면)" ↔ "모니터링 모드(교사 화면)" 전환. 한 사람이 둘 다 쓰거나 두 사람이 나눠 맡음. 학생은 별개. 모니터링 지표(학생엔 비공개, 기본 세팅): ①폴리포니 미터(발산↔수렴 추이) ②참여 현황(제출/미제출) ③군집별 인원·분포 ④개별 응답 추적(닉 기준, 무대엔 익명).

## Q7 — AI 엔진 & 프런트엔드 스택
**Claude:** LLM 제공자(Claude/OpenAI/둘다) + 프런트(React-TS/Next/순수JS)? API 키 보안 수준?
**User:** (1)로 가되 TypeScript 모름 → JS/JSX(No tsx).
**Takeaway:** AI=Anthropic Claude(설정서 API키·모델 선택). 프런트=React+Vite **JS/JSX**(TS 미사용). 캔버스=CSS 열 + SVG 분기선. 백엔드=Firebase. **API키 보안(Claude 결정)**: AI 호출은 관리자 기기 한정, 키는 관리자 브라우저 localStorage에만 저장. 학생/교사는 Firebase의 결과만 구독 → 키 비노출. (배포 시 Cloud Functions로 이전 가능.)

## Q8 — 디자인·UX 톤
**Claude:** 악보 메타포 / 미니멀 화이트 / 다크 네온 / 종이 손그림 중? (추천: 악보+미니멀 절충)
**User:** 추천대로.
**Takeaway:** 차분한 종이 톤 배경 + 동등 채도의 색 가닥(voice)이 수평으로 흐르는 악보/미니멀 절충. 어떤 가닥도 시각적으로 우월해 보이지 않게.

## Q9 — 세션 종료
**Claude:** 누가/어떻게 종료(수동/자동/둘다)? 종료 후 화면(읽기전용/요약/잠금)? 데이터(보존+내보내기/보존/삭제)?
**User:** A:1, B:1·2, C:1.
**Takeaway:** **수동 종료**(관리자 버튼, 정해진 결말 조건 없음 — 대화엔 last word 없음). 종료 시 **읽기전용 회고 모드**(입력 잠김, 전체 전개 스크롤) + **AI 비위계적 voice 요약**(결론 아님, "어떤 차이들이 살아있었나"; 우세 표현 금지 유지). 데이터 = **Firebase 보존 + JSON 내보내기**.

## 확정된 기본 가정 (사용자 확인됨)
- 사이클 상태머신: 대기→응답수집→마감→AI군집화→관리자 군집확정→AI 후속질문 제안→관리자 선택·발행→다음 라운드 (수동 전환).
- 데이터 모델: Session / Round / Opinion / Cluster(부모군집 링크) / Participant / Reaction(옵션).
- 규모: 동시 학생 ~30–40, 라운드 수십 개.
- 언어: 한국어 우선.
- MVP: 핵심 사이클 + 3뷰 + 열 뷰 캔버스 + AI 군집/질문제안 + Firebase 실시간 + 설정(API키). 2순위: 분기(계보)뷰, 학생 반응 모드, AI 회고 요약.
